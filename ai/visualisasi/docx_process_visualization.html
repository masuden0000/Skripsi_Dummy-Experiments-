Visualisasi Proses Pembuatan DOCX (Tipe: Proposal)

Fokus halaman ini adalah alur proses dan peran fungsi. Desain sengaja dibuat sederhana agar mudah dibaca.

1) Cara Menjalankan

Command utama:

python ai/manage.py docx --type proposal --source-doc file.pdf

Default input/output:

  • document_metadata.payload (metadata hasil extractor)
  • data/output_chunks.json (chunk + sumber halaman/header)
  • data/proposal_template.docx (hasil dokumen)

2) Alur Diagram (Flowchart Sederhana)

3) Alur Berurutan (Ringkas)

  1. manage.py::main() -> parse command docx, lalu panggil run_docx(...)
  2. manage.py::run_docx() -> validasi --type proposal, lalu panggil generator.generate_proposal_docx(...)
  3. model_ai/docx/generator.py::generate_proposal_docx() -> orkestrasi: load metadata, load chunks, translate style, render docx
  4. model_ai/docx/metadata_loader.py::load_document_metadata() -> baca document_metadata.payload + validasi schema DocumentMetadata
  5. model_ai/docx/chunk_loader.py::load_chunk_sources() -> baca output_chunks.json lalu normalisasi data chunk
  6. model_ai/docx/style_mapping_pipeline.py::translate_docx_style_config() -> ubah metadata typography/spacing/numbering jadi konfigurasi style DOCX
  7. model_ai/docx/docx_renderer.py::render_proposal_docx() -> set layout + style + halaman awal + body + sumber + save .docx

4) Detail Per File dan Fungsi

Urutan File Fungsi Kunci Tugas
1 ai/manage.py main(), run_docx() Entry point CLI, validasi argumen, trigger generator.
2 ai/model_ai/docx/generator.py generate_proposal_docx() Koordinator alur end-to-end untuk pembuatan DOCX.
3 ai/model_ai/docx/metadata_loader.py load_document_metadata() Muat + validasi metadata agar aman dipakai renderer.
4 ai/model_ai/docx/chunk_loader.py load_chunk_sources(), match_sources_for_section() Siapkan data sumber per bagian dokumen.
5 ai/model_ai/docx/style_mapping_pipeline.py translate_docx_style_config() Terjemahkan style deskriptif ke konfigurasi render.
6 ai/model_ai/docx/docx_renderer.py render_proposal_docx() + fungsi internal render Bangun isi dokumen dan simpan file akhir .docx.

5) Data Mengalir Dari Mana Ke Mana

document_metadata.payload + output_chunks.json -> generator.py -> docx_renderer.py -> file hasil .docx.

Kalau chunk tidak match dengan section, dokumen tetap terbentuk, tapi bagian sumber bisa tampil sebagai Sumber: [BELUM DITEMUKAN].

6) Input - Proses - Output Per Langkah

File::Fungsi Input Proses Output
manage.py::main() Argumen CLI: docx --type proposal --source-doc file.pdf (+ optional --chunks/--output) Parse command, pilih branch docx, terus kirim parameter ke run_docx() Panggilan fungsi run_docx(...) dengan parameter final
manage.py::run_docx() doc_type, source_doc, path chunks, path output Validasi doc_type == proposal, lalu panggil generate_proposal_docx() dengan selector metadata Supabase Eksekusi generator DOCX atau SystemExit jika tipe tidak valid
metadata_loader.py::load_document_metadata() source_doc yang menunjuk row document_metadata Ambil payload dari Supabase, lalu validasi schema ke model DocumentMetadata Objek DocumentMetadata terstruktur
chunk_loader.py::load_chunk_sources() File data/output_chunks.json Baca list chunk, normalisasi field, buang entri tidak valid List ChunkSource[] siap dipakai render
style_mapping_pipeline.py::translate_docx_style_config() DocumentMetadata (typography, spacing, numbering) Konversi nilai deskriptif ke konfigurasi style DOCX (alignment, caps, posisi nomor halaman) Objek DocxStyleConfig
generator.py::generate_proposal_docx() source_doc, path chunks, output path Orkestrasi: load metadata dari Supabase + load chunks + translate style + panggil renderer Path hasil dokumen dari renderer
docx_renderer.py::render_proposal_docx() DocumentMetadata, ChunkSource[], DocxStyleConfig, output path Set page layout/style, render halaman awal & body, cocokkan sumber section, tulis caption, simpan file File .docx final (default: data/proposal_template.docx)
