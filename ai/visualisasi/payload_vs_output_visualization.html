document_metadata sebagai source of truth metadata

Runtime AI sekarang membaca metadata utama dari document_metadata.payload. File lokal, jika masih ada, hanya berperan sebagai cache atau bahan debugging.

Input

`output.md` + `output_chunks.json`

Konten mentah hasil setup. Ini bahan konteks untuk ekstraksi, belum jadi metadata terstruktur.

Proses Extract

`extract_document_metadata()`

LLM + grounding chunk menyusun object `DocumentMetadata` (dengan beberapa heuristik rule-based).

Legacy

File lokal: `ai/data/output.json`

Jika file ini masih ada, perannya hanya sebagai snapshot lokal untuk debugging, bukan source utama runtime.

  • Format: JSON terstruktur.
  • Berguna untuk inspeksi manual saat development.
  • Tidak lagi wajib ada agar command DOCX atau schema-diff bisa berjalan.
Source of Truth

Supabase: `document_metadata.payload`

Disimpan oleh `save_to_supabase(metadata)` ke kolom `jsonb` dan dipakai ulang oleh command AI melalui `source_doc`.

  • Format: nested JSONB dalam 1 kolom `payload`.
  • Satu row per dokumen PDF melalui `source_doc`.
  • Akses field via SQL operator JSON (`->`, `->>`).
So what

Perbedaan sekarang ada di peran, bukan hanya tampilan

Bentuk JSON-nya bisa sama, tetapi runtime utama hanya menarik metadata dari Supabase. Jadi kalau Anda ingin tahu metadata resmi yang dipakai AI, lihat `document_metadata.payload`, bukan file lokal.

select source_doc, payload from public.document_metadata order by extracted_at desc limit 5; select source_doc, payload->'sources' as sources from public.document_metadata order by extracted_at desc limit 5;
