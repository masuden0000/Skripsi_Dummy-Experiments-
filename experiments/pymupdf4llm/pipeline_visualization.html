PyMuPDF4LLM - Diagram Alur File dan Data

Diagram ini memetakan bagaimana file di folder experiments/pymupdf4llm saling terhubung, dari sumber data awal, proses ingest, retrieval RAG saat chat, ekstraksi metadata, schema diff, sampai generate DOCX.

Entry: manage.py Storage: Supabase (document_chunks, document_metadata) Embedding: GoogleGenerativeAIEmbeddings (768 dim) LLM: ChatGroq structured output

1) Setup Pipeline (PDF -> Chunk JSON -> Supabase)

Dijalankan oleh command python manage.py setup (opsi --skip-ingest untuk berhenti di JSON).

Entry manage.py run_setup()
  • Panggil extract_chunks()
  • Panggil upsert_embeddings() jika tidak skip
->
Loader model_ai/loader/pdf_extractor.py get_page_chunks() extract_chunks()
  • Baca file.pdf via pymupdf4llm.to_markdown(page_chunks=True)
  • Tulis markdown mentah ke data/output.md
->
Chunking model_ai/loader/chunk_builder.py build_sections() build_payload()
  • Deteksi heading/BAB, filter noise heading
  • Hitung page range tiap chunk
  • Set relasi chunk_prev dan chunk_next
->
Output Lokal data/output_chunks.json
  • Payload utama untuk ingest dan debugging
->
Ingest DB model_ai/loader/supabase_ingest.py upsert_embeddings()
  • Embed tiap chunk
  • Upsert ke tabel document_chunks
  • Key konflik: source_file,chunk_index

2) Chat RAG Runtime (Frontend -> /api/chat -> Retrieval -> LLM)

Dijalankan oleh command python manage.py ui.

Frontend frontend/index.html frontend/app.js frontend/styles.css
  • User kirim pertanyaan
  • POST ke endpoint /api/chat
->
HTTP Server model_ai/ui/chat_server.py ChatHandler.do_POST()
  • Validasi body JSON
  • Panggil ask_rag(question)
->
RAG Service model_ai/rag/rag_service.py retrieve_chunks() build_chain() ask_rag()
  • Embed query
  • RPC match_document_chunks ke Supabase
  • Bangun context, invoke Groq structured output
->
Response RAGResponse (JSON)
  • answer, bullet_points, steps, citations
  • Status grounding: grounded / insufficient_context / out_of_scope

3) Metadata Extraction Pipeline (Supabase -> output.json + document_metadata)

Dijalankan oleh command python manage.py extract.

Prompt Registry model_ai/extractor/prompts.py prompts/*.md
  • Load template + query + top_k
  • Menentukan retrieval query per key metadata
->
Extractor Core model_ai/extractor/doc_extractor.py extract_document_metadata()
  • Loop KEY_REGISTRY per kategori (typography, layout, numbering, dst)
  • Retrieve chunks dari Supabase lalu expand per header
  • Invoke Groq structured output
->
Schema Types model_ai/extractor/models.py DocumentMetadata
  • Normalisasi field legacy
  • Struktur output metadata final
->
Dual Output data/output.json table document_metadata
  • Simpan lokal ke JSON
  • Upsert payload metadata ke Supabase

4) Schema Diff Pipeline (Baseline output.json vs Free Extraction)

Dijalankan oleh command python manage.py schema-diff.

Baseline model_ai/extractor/schema_differ.py load_old_schema()
  • Baca dan flatten data/output.json
->
Broad Retrieval fetch_broad_chunks()
  • Query luas ke Supabase untuk cakupan aturan dokumen
->
Free Extraction free_extract_all_rules()
  • Prompt bebas dari free_extraction.md
  • Output key-value JSON tanpa batasan schema ketat
->
Diff Output schema_diff_YYYYMMDD_HHMMSS.json schema_diff_YYYYMMDD_HHMMSS.md
  • Kategori: matched, changed, new_fields, removed

5) DOCX Generation Pipeline (output.json + output_chunks.json -> .docx)

Dijalankan oleh command python manage.py docx --type proposal.

Entry manage.py run_docx()
  • Validasi type proposal
  • Panggil generate_proposal_docx()
->
Load Inputs model_ai/docx/metadata_loader.py model_ai/docx/chunk_loader.py
  • Load metadata terstruktur
  • Load sumber chunk per bagian
->
Style Translation model_ai/docx/style_translator_llm.py translate_docx_style_config()
  • Rule-based map
  • Opsional LLM normalization untuk style config
->
Renderer model_ai/docx/docx_renderer.py render_proposal_docx()
  • Set margin/orientation/numbering
  • Render BAB + placeholder + source block
->
Output data/proposal_template.docx
  • Dokumen template siap diedit lanjut

Data Source dan Storage Map

Sumber Utama

file.pdf di root repo adalah sumber awal untuk proses chunking/ingest.

Artifact Lokal

data/output.md, data/output_chunks.json, data/output.json, dan file schema_diff_*.json/.md.

Storage Supabase

Tabel document_chunks untuk retrieval RAG, tabel document_metadata untuk hasil ekstraksi terstruktur.

Command File yang menjadi entry Output utama
manage.py setup model_ai/loader/pdf_extractor.py + model_ai/loader/supabase_ingest.py data/output_chunks.json + document_chunks
manage.py ui model_ai/ui/chat_server.py + model_ai/rag/rag_service.py HTTP chat response JSON untuk frontend
manage.py extract model_ai/extractor/doc_extractor.py data/output.json + document_metadata
manage.py schema-diff model_ai/extractor/schema_differ.py data/schema_diff_*.json + data/schema_diff_*.md
manage.py docx --type proposal model_ai/docx/generator.py data/proposal_template.docx
Heads-up penting: ingest memakai key konflik source_file,chunk_index. Jika Anda ingest dokumen lain tapi tetap memakai nama sumber yang sama, row lama bisa ter-overwrite.

Referensi schema DB: infra/supabase_setup.sql dan infra/supabase_metadata.sql.
